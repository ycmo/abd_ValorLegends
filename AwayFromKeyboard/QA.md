# AwayFromKeyboard (掛機任務) 需求釐清與 Q&A

這份文件由任務管理 Agent 整理，旨在進入進階實作前，與使用者對齊需求、技術依賴與安全處理邊界。請您針對以下問題 (Q1~Q7) 進行回覆，以便後續推進。

## 1. 目前已知狀態與阻塞

*   **已知狀態**：掛機主迴圈骨架（`main.py`）、初步的防呆回主城機制（`ui_recovery.py`）與設定檔已建立。使用者已在本地端手動補齊 `pyperclip` 套件，環境依賴問題已暫時排除。
*   **目前阻塞**：為保持程式碼穩定與安全，目前全面停止實作與修改外部檔案（包含 `requirements.txt`），須待下列需求與邊界釐清，並擬定下一步計畫後再恢復開發。

## 2. 需求上還不清楚的地方

*   **Q1: 任務擴充性**
    第一階段的掛機只排入了「點金手 (Midas)」。未來的掛機排程是否預計會動態加入其他任務（例如：半夜自動掃蕩魔法商店、公會祈願）？
    *(這會影響我是否要把 `run_midas` 抽成更通用的任務佇列機制)*
*   **Q2: 帳號切換的時間節奏**
    目前的規劃是「A -> B -> C -> D」無縫連續切換並做點金手，全部做完後再一次性休眠 7 小時 50 分鐘。是否需要在**每次切換帳號之間**也保留一段強制休息或載入緩衝時間？

## 3. 技術與環境上需要確認的地方

*   **Q3: 切帳號模組的執行反饋**
    現有的 `switch_account.py` 是一個獨立執行的腳本，執行完畢不會回傳 True/False。如果登入過程中發生卡頓或密碼錯誤，主程式無法在第一時間得知失敗，只能靠稍後「是否能看見主城」來盲測。
    **您是否同意我後續提出一份「微調 `switch_account.py` 回傳值」的修改計畫？** (同意/不同意/暫不考慮)
*   **Q4: 依賴套件管理政策**
    您剛才指示「嚴禁修改 AwayFromKeyboard 目錄以外的地方」。關於 `pyperclip` 未被記錄在 `requirements.txt` 的問題，未來您傾向由您手動維護 `requirements.txt`，還是同意由我另行提計畫來幫您更新？

## 4. 安全邊界與失敗處理需要決定的地方

*   **Q5: 單一帳號失敗策略 (Fail-Safe)**
    當某個帳號遇到無法排除的異常（例如：無盡的彈窗、遊戲卡死，導致腳本經過多次嘗試後仍找不到「主城」），您希望腳本的處置方式是：
    (A) **跳過**：放生該帳號，直接強制呼叫 `switch_account` 切到下一個帳號繼續做。
    (B) **中斷**：直接停止掛機程式，留下報錯與截圖，等待您早上起床人工處理。
*   **Q6: 最底層的救援機制 (App 重啟)**
    如果連切換帳號的流程都卡住了（例如卡在 Google 系統畫面），導致後續的圖像辨識完全失效，是否需要為掛機腳本加入「強制關閉遊戲 APK 並重新啟動」的終極救援指令？
