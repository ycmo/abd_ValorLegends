# 疾風的呼喚 (Call of the Gale) 自動化模組

這是一個針對手遊《Valor Legends》中「疾風的呼喚」小遊戲所開發的全自動掛機腳本。

---

## 一、遊戲玩法介紹 (How to Play)

「疾風的呼喚」是一個類似彈珠台的物理碰撞小遊戲。玩家需要消耗「卷軸」進入關卡，發射飛鏢來獲取資源，最後用資源升級角色來通關。

### 1. 發射飛鏢與收集資源
進入關卡後，玩家會在下方看到可發射的飛鏢（即能量）。玩家需要按住飛鏢並往下拉，調整角度後放開，飛鏢就會像彈珠一樣在盤面上彈跳，擊碎障礙物並收集「飯糰」。
![遊戲操作畫面](file:///E:/antigravity/adb_vl/call_of_the_gale/runtime_captures/02_操作.png)

### 2. 升級角色與出發
當畫面上的飛鏢全數發射完畢後，就進入了升級階段。畫面上方會顯示你這回合收集到的「飯糰總數」，而右下方藍色按鈕上方有一圈細長的紫色框，裡面會顯示當前的「升級門檻」。
玩家必須消耗飯糰來升級右側的角色陣容，提升戰鬥力。當飯糰被榨乾（低於升級門檻）後，才能點擊右下角的「出發」按鈕，讓角色與敵人自動戰鬥。
![升級與出發畫面](file:///E:/antigravity/adb_vl/call_of_the_gale/runtime_captures/03_升級出發.png)

### 3. 過場動畫與繼續挑戰
點擊出發後，雙方角色會進入戰鬥過場動畫，此時可以點擊右下角的跳過圖示 `>>|` 來跳過動畫。戰鬥結算後，點擊「繼續挑戰」即可消耗一張新卷軸，進入下一個回合。
![過場動畫](file:///E:/antigravity/adb_vl/call_of_the_gale/runtime_captures/04_過場.png)
![繼續挑戰](file:///E:/antigravity/adb_vl/call_of_the_gale/runtime_captures/05_繼續挑戰.png)

當右上角的「卷軸」數量歸零時，代表挑戰次數已用盡，遊戲結束。
![卷軸結束](file:///E:/antigravity/adb_vl/call_of_the_gale/runtime_captures/06_卷軸結束.png)

## 二、機器人自動化設計 (Robot Automation Design)

為了讓電腦能夠完美重現上述的操作，本模組結合了 ADB 控制、電腦視覺 (Template Matching) 以及光學字元辨識 (EasyOCR)，設計出以下核心機制：

### 1. 動態飛鏢裝填偵測 (最速連發)
因為飛鏢每次在彈珠台裡彈跳的時間長短不一，死板的固定等待時間會嚴重拖慢效率。
- **解法：** 程式會擷取發射台的「空白背景 (`empty_slot.png`)」。當發射台的影像相似度突然降低，代表「飛鏢憑空出現擋住背景了」，此時程式就會立刻拉弓發射，達成毫秒級的最速連發。

### 2. 智慧飯糰消耗與動態升級
遊戲後期的升級門檻會隨著次數飆升，且數值龐大時會使用 `K` (千) 或 `M` (百萬) 作為縮寫，這對自動化是一大挑戰。
- **動態比對：** 程式會透過 OCR 同時讀取畫面上方的「飯糰總數」以及按鈕上方的「升級門檻」。
- **防呆數值轉換：** 內建專屬的解析器 `parse_game_number`，自動將 `10.5K` 轉換為 `10500`，過濾逗號，甚至會將 OCR 容易誤判的字元（例如把英文字母 `O` 修正為數字 `0`）自動校正。
- **榨乾機制：** 最後一發飛鏢落地後，程式會「無條件先長按升級 3 秒」，接著透過迴圈不斷比對：「飯糰總數 >= 升級門檻」就繼續長按，確保戰力最大化後才點擊「出發」。

### 3. 解決模擬器無效點擊 (微小位移長按)
許多 Android 模擬器在接收 ADB 的 `swipe(起點, 終點, 時間)` 指令時，如果起點和終點的座標完全相同，會將其判定為無效點擊或單次短按。
- **解法：** 程式在執行「長按升級」時，會刻意加上 2 pixel 的位移（從 `800, 400` 滑動到 `802, 402`），確保系統 100% 觸發長按機制。

### 4. 強健的場景切換與按鈕捕捉
遊戲中的按鈕（如出發、跳過）時常會有呼吸燈閃爍動畫或被特效短暫遮擋，且 Android 模擬器時常會發生點擊事件遺失（漏點）的狀況。
- **出發按鈕捕捉：** 程式將「出發」的影像辨識門檻放寬至 0.7，以對抗按鈕發光閃爍造成的特徵改變。同時加入了 `has_clicked_depart` 的狀態鎖，確保程式「真的有成功命中且點擊到出發按鈕」後，才會在按鈕消失時判定為成功進入過場，徹底杜絕了「因為閃爍沒看到就誤以為已經點擊」的邏輯漏洞。
- **場景解耦與連點防漏：** 程式將「跳過過場動畫」與「結算繼續挑戰」這兩個不同的場景徹底解耦為兩段獨立的等待邏輯：
  1. **過場跳過階段：** 程式會連續尋找「跳過」按鈕並執行點擊。即使點了也不會馬上停止（防漏點），而是會繼續監控，直到看見下一個場景的「繼續挑戰」按鈕，才確認動畫徹底結束並安心交棒。
  2. **結算挑戰階段：** 專心等待結算畫面，點擊「繼續挑戰」後進入下一輪。

### 5. 多重 Fail-Safe 安全保護
為避免畫面卡頓或 OCR 失誤導致無限循環，所有迴圈都設有跳出條件：
- 若連續 5 次無法辨識飛鏢、飯糰數量，程式會主動中止。
- 升級長按次數超過上限卻仍未降至門檻以下，直接觸發錯誤 (`sys.exit(1)`) 以防卡死。

## 三、程式架構與使用方式

本模組遵循 DRY 原則，將邏輯拆分為兩個主要腳本：

1. **`single_shoot.py` (核心單回合模組)**
   - 負責單個關卡內的完整操作（開場盲狙 -> 平行載入 OCR -> OCR 辨識連發飛鏢 -> 檢查飯糰並長按升級 -> 點擊出發）。
   - **指令：** `.\.venv\Scripts\python.exe call_of_the_gale\scripts\single_shoot.py`

2. **`auto_shoot.py` (全自動大迴圈)**
   - 負責自動周回。它會先檢查「剩餘卷軸」，然後呼叫 `single_shoot.py` 來執行戰鬥。戰鬥結束後自動點擊「跳過」與「繼續挑戰」。卷軸歸零時自動點擊「離開」。
   - **指令：** `.\.venv\Scripts\python.exe call_of_the_gale\scripts\auto_shoot.py`

*(若需觀看除錯影像，可在指令後方加上 `--debug` 參數，截圖與裁切畫面將會存入 `debug_output/` 中。)*
